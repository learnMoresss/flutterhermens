# Flutter Hermes 移动端方案 — 架构计划书

> **结论：方案可行，是当前最现实、工程成本最低的一条路线。**  
> 已从「找现成 App」切换到「做薄壳客户端 + Node 中转」，方向正确。

---

## 一、方案本质

```text
Flutter App
   ↓
Node 转发层（Gateway / BFF）
   ↓
Hermes Web API（与 WebUI 同源接口）
```

这是典型的 **BFF（Backend For Frontend）+ 代理适配层**，等价于：

> 把 Hermes WebUI 的能力 API 化 + 移动端化

---

## 二、关键前提（必须先确认）

> Hermes 有 WebUI → 通常有完整接口，但需验证以下 3 点。

### 1. 是否存在独立 API

检查是否存在类似路径：

```text
/api/chat
/api/session
/api/auth
/ws
/socket.io
```

或：

```text
/v1/chat
```

若 WebUI 为 React / Vue + REST / WebSocket，基本可直接复用。

### 2. Token 是否可复用

设计路径：`登录 → token → 调 API`

需确认：

- token 是否为 JWT
- 是否依赖 session cookie
- WebSocket 鉴权方式

### 3. CORS / Origin 绑定

若 Hermes WebUI 限制 origin 或 referer，**Node Gateway 服务端转发**即可绕过，无需在 Flutter 侧处理。

## 三、标准架构（优化版）

```text
           ┌──────────────┐
           │ Flutter App  │
           │ (UI only)    │
           └──────┬───────┘
                  │ HTTPS
                  ▼
     ┌────────────────────────┐
     │ Node Gateway (BFF层)   │
     │ - auth/token 管理      │
     │ - websocket 转发       │
     │ - SSE 流式转换         │
     │ - 文件上传代理         │
     └──────┬─────────────────┘
            │ internal API
            ▼
      Hermes Web API
```

---

## 四、Node Gateway 职责（只做 4 件事）

### 1. 登录代理

```http
POST /login
```

内部流程：

```text
Flutter → Node → Hermes login → token → Flutter
```

Node 不存业务逻辑，只转发。

### 2. Chat 转发（核心）

```http
POST /chat
```

Node 负责：

- 转发 Hermes WebSocket / HTTP
- 将 SSE 转为 Flutter 可消费的流

### 3. WebSocket 透传（推荐）

若 Hermes 有 WS：

```text
Flutter ⇄ Node ⇄ Hermes WS
```

Node 只做：鉴权、转发，**不解析 payload**。

### 4. 文件上传代理

```http
POST /upload
```

Node 负责：

- 接收 Flutter 文件
- 转发至 Hermes file API
- 返回 URL 或资源 id

## 五、Flutter App 实现建议

> 原则：**尽量用现成的，不造轮子。**

### 技术栈

| 层 | 推荐 |
|---|---|
| Chat UI | `flutter_chat_ui` |
| 状态管理 | `riverpod` / `bloc` |
| HTTP | `dio` |
| WebSocket | `web_socket_channel` |

### 可直接复用的能力

- Markdown 渲染
- Code block
- Message list
- Input bar
- Typing indicator

### 分层结构

```text
UI 层（纯展示）
   ↓
API Client（对接 Node Gateway）
   ↓
Hermes 能力（经 Gateway 间接使用）
```

**Flutter 不直接碰 Hermes 逻辑。**

## 七、必须注意的坑

### 坑 1：WebSocket 长连接

Hermes 可能不支持断线重连，或 session state 绑定在 WS 上。

**对策：** Node 必须实现断线重连 + session 恢复。

### 坑 2：Stream 格式不标准

**对策：** 统一转换链路：

```text
Hermes stream → OpenAI stream → Flutter stream
```

否则 Chat UI 难以维护。

### 坑 3：Token 生命周期

**对策：** 实现 refresh token（若有），或 session 过期后自动 re-login。

### 坑 4：文件权限

**对策：** 弄清 Hermes 文件是公开 URL 还是 internal id，Gateway 统一对外格式。

Hermes WebUI → Mobile BFF Layer → Flutter Client

即 **AI Agent Mobileization Layer**，与 ChatGPT App、Claude Mobile、OpenAI mobile SDK 同类架构。

## 九、关键建议：协议标准化

Node Gateway **必须做协议标准化**，对外统一为：

```text
OpenAI-compatible API
```

收益：

- 未来可换 Hermes
- 可接 Claude Code、Codex、MCP、本地模型
- **Flutter 完全不用改**
